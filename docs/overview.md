# Claude Workflow System — Overview

## Project Overview

This repo contains the design and implementation plan for a complete, integrated Claude AI development workflow system. The system covers every phase of development — from planning and codebase exploration through implementation, testing, and tracking — using Claude Code as the primary interface.

The workflow gives a developer: queryable codebase intelligence (so Claude knows what exists without reading the whole project), standardized git and Jira operations via dedicated agents and skills, an automated TDD testing framework, and a planning protocol where the plan doc is the single source of truth — Jira and TODO.md are downstream views, not co-equal tracking layers.

The system is organized into 9 sub-plans, each covering one subsystem. Sub-plans are designed to be worked in dependency order and implemented once the design phase is complete. Sub-plan 09 (Developer Setup) is the final deliverable — a script that installs everything built in 01–08 onto a new machine in one step.

## How the Workflow Works

```
┌─────────────────────────────────────────────────────┐
│                  Planning Layer                     │
│  (Plan Mode → Plans → Architecture Understanding)   │
└─────────────┬───────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────┐
│                 Execution Layer                     │
│  (Research → Verify Values → Write Code → Test)     │
└─────────────┬───────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────┐
│                  Tracking Layer                     │
│   Plan Doc (truth) → Jira (external) / TODO.md      │
└─────────────────────────────────────────────────────┘
```

- **Planning Layer:** Plan mode produces plan docs; architect agent reviews before exit; test strategy agent gates implementation; L-sized work gets a local plan doc committed to the repo.
- **Execution Layer:** The `/infra-init` skill builds a codebase graph and local MCP tools so Claude navigates the project with minimum context cost — returning structured symbol/caller/env-var results on demand rather than grepping raw files. Run on any repo you plan to do meaningful development work in. All infrastructure values are looked up from authoritative sources before use.
- **Tracking Layer:** The plan doc is the single source of truth. Jira is seeded from it at ticket-creation time; updates flow in as comments. TODO.md is a pointer registry — one entry per active plan (plan doc path + Epic key) — not a copy of task detail. No three-way sync is required: marking a task done in the plan doc is the authoritative record.

## Sub-Plans Index

| # | Name | Purpose |
|---|------|---------|
| 01 | Codebase Knowledge System | Generates a queryable codebase graph and MCP tools via the `/infra-init` skill — so Claude knows what exists without reading the whole codebase |
| 02 | Jira Integration | Defines how Jira works as a background tracking layer — tickets for plan-execution, bugs, human tasks, and cross-repo dependencies |
| 03 | Testing System | TDD-first testing framework: `/e2e-init` bootstraps TDD for repos with no tests (unit/integration only — E2E deferred); test-strategy agent defines contracts before code; test-builder writes failing tests from spec; pipeline gating, pre-commit hooks, and runtime observability |
| 04 | Git Workflow | Standard branching, commits (conventional commits), squash-merge, PR review, release tagging, and pre-flight checks via the `git-manager` skill |
| 05 | Agent Architecture | Design specs for every agent in the system: architect (informed isolation), researcher (codebase + infrastructure lookups via appropriate MCP), infra-init agents, integration-engineer (cross-repo via local MCPs), test-strategy, test-builder, jira-workflow-manager. `todo-manager` migrated to `plan-management` skill |
| 06 | Plan Management | Protocol for making and maintaining plans: research-first, architect gate, test strategy gate, plan doc structure, session handoff, archival. Plan doc is the single source of truth; TODO.md is a pointer registry, not a task copy |
| 07 | Rules System | Minimal instruction system: 2 always-on files (CLAUDE.md + filesystem-efficiency.md, ~110 lines total) + 2 path-scoped rules (planning.md, new-repo-setup.md). Detailed guidance lives in agent/skill definitions, not always-on rules |
| 08 | Skills System | All skills in `~/.claude/skills/` — `git-manager`, `infra-init`, `e2e-init`, and `plan-management` (absorbs `todo-manager` agent) |
| 09 | Developer Setup | Sequenced implementation guide and setup script that installs all agents, skills, rules, and MCP servers on a new machine via symlinks (not copies) so repo updates propagate immediately |

## Implementation Order

Sub-plans are implemented in dependency order. Sub-plan 09 (Developer Setup) is the final deliverable — it bundles everything built in 01–08 into a single idempotent install script.

```
Infrastructure as Code (01)
    ↓
Jira Integration (02) ←→ Testing System (03)
    ↓                        ↓
Git Workflow (04)            ↓
    ↓                        ↓
Agent Architecture (05) ←───┘
    ↓
Plan Management (06) [depends on all above]
    ↓
Rules (07) + Skills (08) [parallel — depend on 01–06]
    ↓
Developer Setup (09) [final — installs everything]
```

**Key insight:** Codebase Knowledge System (01) is implemented first, as other systems depend on reliable value lookups and codebase navigation.

## Key Principles

1. **Value verification first** — Always look up, never guess
2. **JIRA is tracking, not workflow** — Work proceeds independently
3. **Plans organize work** — Small plans stay global, large plans go to repos
4. **Tests before code** — Test-builder writes failing tests from the spec; implementation is written to make them pass (red-green-refactor)
5. **Research is delegated via MCP** — The `researcher` agent handles lookups using whatever MCP fits the question: local codebase MCP (symbols, callers, env vars) for codebase questions, AWS MCP for infrastructure values, etc. Grep/Read stay in main context only when code comprehension or judgment is needed
6. **Health checks prevent failures** — Pre-flight checks before risky operations
